# Evaluation Methodology for an Open-Source Context Management System for Streaming AI Agents

*Committee-ready evaluation note. Status: this note fixes the "evaluation is vague" feedback by specifying a layered framework with exact, implementable metrics. It assumes the RQ2 reframe (`research/RQ2-anomaly-detection-reframe.md`) from a globally-supervised `p(urgent)` classifier to a per-entity, label-free anomaly/deviation router (Cloudflare-inspired), with deterministic rules as a hard floor and the contextual bandit demoted to sensitivity-threshold control.*

---

## 1. Evaluation philosophy

The system is a routing layer between heterogeneous event sources and stateless LLM agents. The supervisor's two critiques — (a) the fast/batch framing should mirror Cloudflare's *unsupervised, per-entity anomaly detection* rather than supervised urgency classification, and (b) the evaluation must state *how* it works and *exactly which* metrics it measures — drive a **three-layer** evaluation. Each layer answers a distinct question and is measurable independently, so a failure can be localized.

- **L1 — Routing/anomaly-detection quality (offline, replayed).** *Question: when the router fires "FAST", is the event actually one that warranted immediate push, and is it detected early?* This is a **detection** problem (rare positives, temporal structure), evaluated with imbalance-aware and latency-aware anomaly-detection metrics on a replayed trace with constructed ground truth.
- **L2 — Online behavior under drift and load (streaming, prequential + systems).** *Question: does the router stay correct, calibrated, and economical as the stream drifts, and does the pipeline meet latency/throughput SLAs under that load?* This couples ML-under-drift metrics (prequential, regret, calibration, drift recovery) with streaming-systems metrics (tail latency, throughput, lag, backpressure, recovery).
- **L3 — End-to-end agent task utility (closed-loop).** *Question: does correct routing make the downstream agent more useful — right intervention, right time, at acceptable token cost — and does push delivery beat polling?* Evaluated with timing precision/recall + timeliness, LLM-as-judge content quality, token-waste economics, and a fair push-vs-pull benchmark.

**Central hypotheses (pre-registered):**
- **H1 (anomaly reframe works):** the per-entity deviation router + rules floor achieves higher VUS-PR and affiliation-F1 than (i) a global supervised `p(urgent)` classifier and (ii) a rules-only baseline, on the same replayed trace. *(Answers critique a.)*
- **H2 (engine value is burst-specific):** the context engine's advantage in mean asymmetric reward and timing-F1 over a no-engine baseline **shrinks as burstiness B → 0**, demonstrating the value is specifically in handling bursty streams.
- **H3 (SLA + economy):** the FAST/BATCH split holds FAST-lane p99 event-time latency-to-agent under the SLA at sustainable throughput, and reduces token-waste rate vs an always-FAST baseline.
- **H4 (push beats poll):** push delivery dominates polling on the latency/cost/staleness Pareto across all swept poll intervals.

## 2. Benchmark / dataset

It is cited as a genuine external **metric template and burstiness reference**, *not* as validation of this thesis's downstream-feedback labelling scheme. **Caveat:** verify access licence (de-identified content public; raw screenshots may be restricted) before redistributing data.

**Real, reusable benchmarks.**
- **ProactiveBench / "Proactive Agent"** (Lu et al., arXiv:2410.12361; repo `github.com/thunlp/ProactiveAgent`, **Apache-2.0**) — 6,790 events (Coding 2,275 / Writing 2,354 / Daily Life 2,161) with a reward-model-as-judge (reported F1 ≈ 0.918). Best-licensed directly reusable proactive-agent benchmark; use its 2×2 {need}×{proposed} confusion structure and reward-model-as-judge pattern. *(Org is `thunlp`, not `THUDM`.)*
- **Memory benchmarks** for the context-engine claim: **LongMemEval** (Wu et al., ICLR 2025, arXiv:2410.10813, MIT licence; 500 questions, S/M variants up to ~1.5M tokens) and **LoCoMo** (Snap Research; up to 35 sessions) — used to justify Graphiti's temporal KG over flat RAG (verify LoCoMo's licence in-repo before redistribution).
- **Streaming-systems harness templates:** **Nexmark** (query/resource model), **OpenMessaging Benchmark** (driver/worker + latency-percentile reporting), and the **Karimov sustainable-throughput** framework. Cite **YSB** explicitly as the negative example (Redis bottleneck, untuned defaults) to justify departures.

**Bursty event-stream trace with MEASURED burstiness.** Because no canonical bursty-routing benchmark exists, constructing one is itself a contribution. Procedure:
1. **Record** a real trace from the three source families (Slack, JIRA, e-commerce/observability telemetry) into the Iceberg/Paimon log, preserving original event-time timestamps and CloudEvent `subject` (entity key).
2. **Measure burstiness** per source and per entity using the parameter **B** (Goh & Barabási; finite-sequence estimator Kim & Jo, *Phys. Rev. E* 94:032311, 2016): `B ∈ [−1 (regular), 0 (Poisson), +1 (maximally bursty)]`, computed from the coefficient of variation of inter-event times, plus the memory coefficient M. Use the `bursty_dynamics` package. Report measured B alongside ProAgentBench's B = 0.787 for external validity.
3. **Replay** deterministically from the Iceberg/Paimon store (a Kappa+ capability a stateful supervised model cannot match) so every condition sees the identical stream.

**Ground-truth urgency labels (and why this is hard).** "True urgency" has no clean, independent label — it is only revealed by *delayed, selection-biased, policy-dependent* downstream agent verdicts (the design's Achilles heel). The router only ever observes the lane it chose, so the batch lane is never counterfactually observed (missing-not-at-random). We therefore use a **three-tier label strategy**:
- **Silver labels:** downstream verdicts {appropriate, too_urgent, too_slow, irrelevant}, treated as noisy/delayed.
- **Injected oracle:** a known subset of events deliberately constructed as urgent/anomalous, with **hard negatives** (need-less moments contextually similar to need moments, per ProAgentBench) so precision is not inflated by trivial idle periods. This gives unbiased precision/recall/AUPRC.
- **Gold set:** a stratified human-adjudicated subset (oversampling boundary, too_urgent, too_slow cases) defining the optimal intervention point `t_optimal` and content quality; report silver-vs-gold agreement so reviewers can gauge label noise. Explicitly state which definition of `t_optimal` is used (first human/agent action timestamp vs SLA-derived vs annotator consensus), since each yields a different lead-time zero.

## 3. L1 — Fast-path routing as anomaly detection: metrics

The fast path is scored as **time-series anomaly detection**, not point-wise classification.

**PR-AUC vs ROC-AUC (imbalance).** Urgent events are rare, so **ROC-AUC is misleading**: `FPR = FP/(FP+TN)` and the dominant TN count keeps FPR (hence ROC-AUC) optimistically near 1 even with many false alarms. **PR-AUC** (precision vs recall, ignores TN) is the imbalance-aware scalar; **always report it against the prevalence baseline** (random PR-AUC = positive prevalence). State prevalence per source in every table, since it differs sharply across Slack/JIRA/telemetry.

**precision@k.** Precision among the top-k highest-scored events — the operationally relevant regime where agents can only act on a budget of k alerts/window.

**Range-based precision/recall (Tatbul et al., NeurIPS 2018).** Generalizes point P/R to *ranges*, decomposing each match into Existence, Overlap-size, Position bias (front/middle/back — lets us reward *early* detection), and Cardinality (penalizes fragmented detections). Requires choosing bias functions, so report as a *sensitivity check* and pair with a parameter-free metric.

**Affiliation precision/recall (Huet et al., KDD 2022).** **Parameter-free** and PA-resistant: scores by the average temporal distance from each prediction to the nearest true event (and vice-versa), normalized against a random predictor (0 = random, →1 = perfect). This is the **primary event-level metric** because it cannot be gamed by a single lucky in-segment hit.

**VUS — Volume Under the Surface (Paparrizos et al., VLDB 2022).** Adds a buffer of length *l* around each true anomaly with a label decaying from 1 to 0, then integrates range-based PR/ROC over all *l* — removing **both** the threshold **and** the window-length choice. **VUS-PR is the single headline number** (imbalance-aware + robust to boundary misalignment).

**Detection latency + NAB scoring.** Define an anomaly window around each true event; only the **earliest** true positive per window is credited, rewarded by a **scaled sigmoid** of its relative position (earlier → higher; right edge → 0) — *not* a linear decay. FPs outside windows carry negative weight; missed windows are negative FNs; the raw score normalizes to **0–100**. Use the **`reward_low_FN_rate`** profile (weights `{fn: 2.0}`) to encode *missed-urgent ≫ false-alarm*. **Also report raw median and p95 detection delay in seconds**, because NAB's normalization hides absolute latency.

**Cost-sensitive recall@low-FPR.** Report recall at a fixed, low false-alarm operating point (e.g. recall@FPR≤1%), mirroring Cloudflare tuning detection rate subject to an acceptable false-positive rate, and total expected cost `= C_FN·FN + C_FP·FP` with `C_FN ≫ C_FP`, swept over the ratio with the cost-minimizing operating point reported.

> **⚠ Pitfall — naive point-adjusted F1.** **Do NOT report point-adjusted (PA) F1 as a headline metric.** Kim et al. (AAAI 2022) prove that PA — flipping an *entire* ground-truth segment to "detected" from a *single* flagged point — inflates F1 so badly that a **uniformly random anomaly score becomes state-of-the-art**, and an untrained model is competitive once PA is forbidden. **What to report instead:** VUS-PR + affiliation-F1 as primary. If PA numbers are needed for comparability with prior work, report the **PA%K curve** (F1 vs K over K ∈ [0,100], or its area, via the `tadpak` library) — never a single K=0 point — with a one-line Kim-et-al. caveat. **Sanity check:** run a uniform-random scorer and a trivial "always-flag" detector through the *entire* suite; any metric that ranks them near the real model is broken for this setting and must be dropped.

**Recommended L1 metric set.**
- **Primary:** VUS-PR (target ≥ 0.6 at the trace's prevalence; report prevalence); affiliation-F1 (target ≥ 0.70).
- **Secondary:** PR-AUC + precision@k; NAB score under `reward_low_FN_rate` (target ≥ 60/100); recall@FPR≤1% (target ≥ 0.80 on injected-urgent); median detection delay (target within the per-source `too_slow` budget).
- **Reference floor the learned router must beat:** a Cloudflare-style rolling robust z-score (median+MAD, current 5-min vs trailing 4–6h window, fire on |z|>3.5 **AND** an absolute floor).

## 4. L2 — Online routing under drift & system load: metrics

Evaluated **prequentially** (interleaved test-then-train) on the replayed stream, **not** a static split.

**Prequential accuracy with fading factor.** Each event is first tested, then trained on. Use the memoryless fading factor: `S_i = x_i + α·S_{i−1}`, `B_i = 1 + α·B_{i−1}`, error `E_α(i) = S_i/B_i`, with α ∈ {0.99, 0.999} stated explicitly. Because FAST/BATCH is imbalanced and bursts are autocorrelated, **report prequential AUC** (imbalance-invariant, doubles as a drift indicator) and **Kappa-temporal** = `(p0 − p'_per)/(1 − p'_per)` against the persistent/no-change classifier — a value ≤ 0 means the router is no better than repeating the last label. Drop bare accuracy as a headline.

**Cumulative regret vs oracle routing.** `regret(T) = Σ_t [ r*(x_t) − r_chosen(x_t) ]`, where the oracle action is reconstructed from logged "appropriate" verdicts. Plot regret vs T against baselines (always-FAST, always-BATCH, rules-only, random, ε-greedy); expect sublinear growth in stationary segments and a recoverable kink at injected drift. State and own the oracle-bias from delayed verdicts.

**Off-policy / counterfactual evaluation.** For offline router comparison on logged data, use the **Li et al. replay (rejection-sampling) evaluator** — unbiased when logging was uniform; cost ≈ K·T retained events (K=2 arms). Because the real logging policy (rules + bandit) is **non-uniform**, also report the **Doubly-Robust (DR)** estimator (consistent if *either* propensity *or* reward model is correct, lower variance than IPS) with IPS as a sanity baseline. **Log per-decision action propensities at serving time now** so this is feasible later.

**Calibration.** Validate the "calibrated `p(urgent)`/score" claim with **ECE** = `Σ_m (|B_m|/N)·|acc(B_m) − conf(B_m)|` (fixed M bins, e.g. 15), a **reliability diagram**, and the **Brier score** (proper scoring rule; ECE is not). Track all three **prequentially** so post-drift miscalibration and online-recalibration recovery are visible.

**Drift detection delay & recovery.** Inject known drift points; measure **detection delay** (instances from true onset to ADWIN/DDM alarm), **false-alarm rate** on stationary windows, and **recovery time** (instances until prequential AUC returns within tolerance of its pre-drift level).

**System metrics (the same event log, open-loop load).** Avoid **coordinated omission**: the replayer injects events on a fixed schedule (wrk2/HdrHistogram methodology), stamps each with its intended arrival time, runs on hardware separate from the system under test, and defines `latency = delivery_time − intended_arrival_time`.
- **End-to-end event-time latency-to-agent**, p50/p95/**p99/p99.9**/max (never just the mean), where `t0 =` CloudEvent event-time arrival and `t_deliver =` commit of the (event, Context Unit) tuple to the entity-partitioned delivery topic such that it is consumable; **decomposed** into ingest+adapter / route / context-assembly (Graphiti+FalkorDB) / delivery. Report FAST-lane and BATCH-lane separately.
- **Sustainable throughput** (Karimov): sweep input rate, report the *knee* where p99 event-time latency stops being flat — not a peak-firehose number.
- **Consumer/watermark lag:** Kafka `records-lag-max`; Flink event-time (watermark) lag.
- **Backpressure / saturation:** Flink `backPressuredTimeMsPerSecond`, `busyTimeMsPerSecond` (target <700/1000 at peak for headroom).
- **Checkpoint & recovery:** `LastCheckpointDuration`; **recovery time** = wall-clock from an induced TaskManager kill until the job re-reaches its pre-failure watermark/lag (validates the Kappa+ replay claim).

## 5. L3 — End-to-end agent task utility: metrics

**Timing prediction + timeliness mapped to the verdict vocabulary.** Build the 2×2 confusion over {true-need, no-need} × {intervened, silent}: Correct-Detection = **appropriate**, False-Detection = **irrelevant**, Missed-Need = **too_slow** (omission). Report Intervention **Precision** (= alert-fatigue proxy), **Recall** (= manual-seek-help proxy), **F1**, plus an over-trigger/False-Discovery rate — matching ProAgentBench/ProactiveBench. **Do not collapse timing into point-wise F1.** For each true need, compute a **signed lead time** `Δt = t_intervene − t_optimal` within a pre-registered tolerance window `[−τ_early, +τ_late]`: inside = appropriate; before `−τ_early` = **too_urgent**; after `+τ_late` (or never) = **too_slow**. Report (a) **median signed lead time + IQR**, (b) a latency-aware event-F1 (NAB sigmoid / PATE proximity weighting), and (c) the full **verdict histogram** as the primary diagnostic. **Asymmetric reward:** a cost matrix grounded in HCI interruption-cost research — reward *appropriate*; penalize *too_urgent* by interruption/resumption cost (Adamczyk & Bailey, CHI 2004); penalize *too_slow* by decayed value-of-information (Horvitz bounded deferral); penalize *irrelevant* most (interruption cost for zero info). Report **mean asymmetric reward per event** (the pre-registered primary L3 metric, also what the bandit/controller optimizes) and keep the raw histogram so reviewers can re-weight.

**Assist-content quality via LLM-as-judge (bias-controlled).** Score **Helpfulness** and **Relevance** via **pairwise** judging with **randomized/swapped order** (count a win only if stable under both orders, cancelling position bias), and **Faithfulness/Groundedness** via RAGAS-style entailment against the Context Units the agent received (catches hallucination). Controls: judge from a **different model family** than the generator, temperature 0, rubric + chain-of-thought, reference-guided where a gold assist exists. **Human spot-check:** label a stratified subset (oversampling boundary/too_urgent/too_slow), report **Cohen's κ** (judge-vs-human and human-vs-human); trust the judge only on slices where κ ≥ ~0.6. Frontier judges fail 50%+ of bias stress tests, so this validation is mandatory.

**Token-cost / "token waste."** Count `prompt_tokens` and `generation_tokens` per event (vLLM exposes `vllm:prompt_tokens_total` / `vllm:generation_tokens_total`). Define **token waste counterfactually**: for each event routed to FAST whose verdict ∈ {too_urgent, irrelevant}, `waste = tokens_charged_on_fast_path − its amortized share under the BATCH-aggregated policy`. Aggregate to `token_waste_rate = wasted_tokens / total_tokens` and `wasted_$` under a **frozen cost model** ($/M input + $/M output, model tier, fixed per experiment). Report **input-token waste separately** — assembled-context input tokens dominate mis-routing cost. This converts the noisy "true urgency" label problem into a measurable economic quantity. Pin the BATCH aggregation policy per experiment (the amortized share depends on it).

**Head-to-head delivery benchmark (pull/poll vs push).** Fairness controls: feed **both arms the identical trace at the identical CO-corrected rate**, with identical agent, broker, partition count, and hardware; **sweep the poll interval** (e.g. 1s / 5s / 30s) and report the **latency/cost/staleness Pareto** — a single interval is cherry-picking. Attribute the poll arm's extra latency explicitly to interval-induced staleness vs empty-poll waste, and the push arm's cost to delivery/connection overhead. Report per-arm p50/p95/**p99/p99.9** latency-to-agent plus throughput and cost. (This is the RTCE-pull vs Kafka-push comparison from `proposal.md` RQ4.)

## 7. Statistical rigor

Because all conditions run on the **same stream**, every comparison is **paired**.
- **Binary timing decision (intervene vs silent):** **McNemar's test** on the paired discordant cells.
- **Continuous metrics** (reward, signed lead time, VUS-PR, faithfulness, latency): **paired BCa bootstrap 95% CIs** + a **sign-flip permutation test** on per-seed deltas, across **≥3 seeds**.
- **Multiple comparisons:** when reporting several conditions/metrics jointly, apply Holm–Bonferroni correction and pre-register the **primary metric** (L1: VUS-PR; L3: mean asymmetric reward) to avoid metric-shopping.
- **"Significantly better"** = the **paired difference CI excludes zero** **AND** the effect is practically meaningful (report the delta and its CI, never bare "SOTA").
- **Label-noise robustness:** jitter ground-truth timestamps and flip a fraction of verdicts; show VUS-PR/affiliation metrics remain stable (they already give near-in-time partial credit) — this directly defends the labels Achilles heel.

## 8. Context-Retrieval (A) and Burst + Scalability (B) — the two added evaluation problems

*The supervisor identified two distinct evaluation problems beyond L1–L3. **(A) Context-retrieval quality** — does the engine assemble the *right* `{entity_state, recent_events, relevant_facts}` envelope? — is the **Lc** layer in §8.7, grounded in ContextBench. **(B) Burst + scalability** — do the routing (RQ2) and delivery (RQ4) designs hold **under bursts at scale**, not on a flattened Poisson load? — is §8.1–8.6 + the matrix in §8.8, grounded in the stream-processing-benchmarking literature. This section adds three pre-registered claims (H5–H7) and three experiments (E9–E11). The two problems intersect at saturation (H7, §8.7): the thesis target workload is bursty (ProAgentBench measured **B = 0.787** [1]).*

### 8.1 The two claims under test

The note's existing H3 ("the FAST/BATCH split holds FAST-lane p99 under the SLA at sustainable throughput") is the steady-state version of two stronger, falsifiable burst-and-scale claims:

- **H5 (it scales, partition-natively).** Scalability is reported **not** as a single peak-throughput number but as a **resource-demand function** [35]: the minimal `{Kafka partitions, Flink parallelism}` needed to *sustain* a load intensity grows **at most linearly** in load and in entity cardinality (distinct CloudEvent `subject`s). The mechanism is that per-entity anomaly state (§4.5 of the RQ2 reframe) shards on `subject` = the Kafka partition key = the Flink `keyBy`, so the scaling axis and the state-growth axis are the same axis [16].
- **H6 (it protects the urgent path under a burst).** During a traffic burst that drives offered load *past* the sustainable-throughput knee, the two-lane design keeps FAST-lane delivery of **genuinely-urgent** events (injected-oracle anomalies + rules-floor events, §2) within `SLO_fast` at **p99 *and* p99.9**, whereas a single undifferentiated lane lets the urgent tail collapse with the aggregate. This is the streaming analogue of priority-aware overload control (DAGOR [37], Breakwater [38], SEDA [39]); its magnitude is **burst-specific** — the two-lane advantage must shrink toward zero as **B → 0** (else the gain is free headroom, not burst protection), mirroring H2's logic.
- **H7 (the right context, even under saturation).** The assembled envelope `C(e)` has high context-recall and context-precision against a per-event gold context `G(e)`, and this quality **degrades gracefully** (sheds optional facts to hold the latency budget) rather than collapsing as the engine saturates during a burst — the brownout trade-off [40] applied to context assembly.

H5–H6 are the **burst + scalability** problem (B); H7 is the **context-retrieval** problem (A), measured on the *same* `(event, envelope)` pairs so the two stories compose: *right event, fast* × *right context, complete*.

### 8.2 Bursty workload: model, generation, measurement

**Why bursts, not Poisson.** Real network and application traffic is **self-similar / long-range-dependent** — bursts exist at every time scale and aggregation does not smooth them out [41, 42]. Poisson "grievously underestimates the burstiness of traffic" [42]; the Poisson-flat case is therefore the **null baseline** (B → 0, Fano → 1, Hurst H = 0.5), not the realistic load. Web/application self-similarity is caused by **heavy-tailed ON/OFF user behaviour** [43], which is exactly the per-entity structure the per-entity router exploits — so the generator must *preserve* it.

**Measuring burstiness (report for every generated condition and the recorded real trace).**
- **Primary scalar — finite-size-corrected burstiness `A_n`** (Kim & Jo, *Phys. Rev. E* 94:032311 [5]). The raw Goh-Barabási burstiness `B = (σ − m)/(σ + m) = (r − 1)/(r + 1)` (with `r = σ/m` the coefficient of variation of inter-event times τ, `B ∈ [−1 regular, 0 Poisson, +1 bursty]` [36]) is **strongly biased for short sequences**: for finite `n` inter-event times the bursty limit `+1` is unreachable, so an uncorrected **per-entity** `B` reads systematically too low. Use `A_n(r) = (√(n+1)·r − √(n−1)) / ((√(n+1) − 2)·r + √(n−1))`, `n` = number of inter-event times, which satisfies `A_n = −1` (regular), `0` (Poisson), `+1` (extreme-bursty) and → raw `B` as `n → ∞` [5]. **Report raw `B` too, only for external comparability with ProAgentBench's 0.787** [1]; state explicitly whether the headline number is **per-entity `A_n` pooled** or `A_n` of the **global aggregate** stream (aggregation lowers apparent burstiness — pick the level that matches ProAgentBench's reporting and say so).
- **Memory coefficient `M`** — the lag-1 Pearson autocorrelation of consecutive inter-event times [36]. `B`/`A_n` is blind to ordering; `M` captures whether long gaps follow long gaps (clustered pace). Report the **(`A_n`, `M`) pair** per source so both burst mechanisms are characterised, not one scalar. Compute the `bursty_dynamics` package [6] which already encodes `A_n` and `M`.
- **Multi-scale cross-checks** (a single scalar hides the scale structure). **Index of dispersion for counts / Fano factor** `F(T) = Var[N(T)]/E[N(T)]` over window `T`: `F(T) = 1` ∀T for Poisson, and **grows with `T`** for bursty/self-similar traffic — plot `F(T)` vs `T` log-log [44]. **Hurst exponent `H ∈ (0.5, 1)`** estimated by **three** methods (wavelet/Abry-Veitch primary, variance-time plot, R/S) with their agreement reported; `H > 0.5` confirms genuine long-range dependence beyond what `A_n` shows [45]. **Peak-to-mean ratio** (max windowed rate / mean rate) as a blunt operational scalar.

**Generating a replayable bursty stream at a target B.** Build the aggregate as a **superposition of independent per-entity Pareto ON/OFF sources** [43, 36]: each CloudEvent `subject` is one heavy-tailed ON/OFF source; their superposition reproduces realistic aggregate self-similarity while keeping per-entity arrival structure intact (required by the per-entity HBOS/robust-z router). The **single tunable knob is the ON-period tail index α**, mapped to target Hurst by `H = (3 − α)/2` for `1 < α < 2` (α = 1.8 → H = 0.6; α = 1.6 → H = 0.7; α = 1.4 → H = 0.8; α → 2 → Poisson, H = 0.5, B = 0). Because `H = (3 − α)/2` is an asymptotic (many-sources, long-horizon) result, **calibrate empirically**: simulate the aggregate, compute the realized finite-size-corrected `A_n`, and binary-search α (and the ON/OFF mean-duration ratio) until `A_n` hits each target on the sweep **B ∈ {0 (Poisson, α → 2), ≈ 0.3, 0.787 (ProAgentBench), ≈ 0.9}**. **Bound α ∈ (1, 2) and truncate the Pareto tail** (α ≤ 1 makes the mean arrival rate ill-defined and the throughput sweep unstable); document the truncation point since it moves both `H` and realized `B`. Freeze the realized timestamps into the Iceberg/Paimon log so every routing/delivery condition replays the **identical** stream (paired comparisons; this is the same E4 burstiness-ablation design as §6). MMPP / ON-OFF (interrupted-Poisson) is the standard alternative generator [36]; prefer Pareto ON/OFF for its single direct knob, and keep an MMPP or block-shuffle generator only if `M` (correlation) must be swept independently of `B` (dispersion).

**Flash-crowd transient.** Steady-state Pareto-ON/OFF gives realistic background burstiness; the supervisor's *acute-surge* concern additionally needs a **flash-crowd** layer [46] — a ramp-plateau-decay intensity multiplier concentrated on a small set of **hot** entity keys, layered over the baseline. This simultaneously stresses (a) RQ2 per-entity scoring on the hot key, (b) Kafka partition hot-spotting under `subject` partitioning, and (c) RQ4 consumer-group lag / cooperative-sticky rebalancing — the precise interaction the burst experiment must expose.

### 8.3 Scalability methodology (Theodolite-style)

Adopt **Theodolite** [35, 16] as the primary scalability design: scalability is a **function** relating provisioned resources to sustainable load, with two dual metrics over a load set L, resource set R, and SLO set S (each SLO a boolean `slo_s: L×R → {true, false}`):
- **Resource-demand metric** `demand(l) = min{ r ∈ R | ∀ s ∈ S: slo_s(l, r) = true }` — the minimal resources to sustain load `l`. **This is the headline RQ2+RQ4 result.**
- **Resource-capacity metric** `capacity(r) = max{ l ∈ L | ∀ s ∈ S: slo_s(l, r) = true }` — the dual.

> **Citation precision.** The 2021 *Big Data Research* Theodolite paper [35] defines a single demand-style metric ("required number of instances per workload") and adopts Karimov's sustainable-throughput concept; the explicit **demand-*and*-capacity pair** is formalised in the 2022 companion (*Empirical Software Engineering* 27) [16]. Cite **both** when quoting the paired definition.

**Workload dimensions to sweep** (Theodolite stresses that "load" must be made precise — 1000 sources at 1 msg/s scales differently from one source at 1000 msg/s):
1. **Message frequency** (events/s) — the primary load axis; x-axis of the demand curve.
2. **Number of distinct keys** (CloudEvent `subject` cardinality) — directly tests whether per-entity anomaly state and `subject`-partitioned delivery scale with **entity cardinality**, the dimension Theodolite shows matters most for keyed/stateful jobs and the one H5 lives on.

**Resources on the y-axis:** the pair `{Kafka partition count, Flink parallelism / TaskManager slots}`. Produce **one demand curve per condition** (FAST lane, BATCH lane, undifferentiated single-lane baseline) and read off whether scaling is **linear, sub-linear, or hits a non-scaling critical point**.

**Sustainability SLO (exactly Theodolite + Karimov).** A `(partitions, parallelism)` configuration **sustains** a load iff **both**: (a) the **consumer-lag-trend** slope — linear regression on Kafka `records-lag-max` over the measurement window — is below a threshold (Theodolite default **2000 records/s** [35]; **empirically recalibrate per source** for Slack/JIRA/telemetry relative to each source's mean rate, do not copy the IIoT default), **and** (b) the trend slope of **p99 event-time latency-to-agent** is ≈ 0 (not continuously rising; Theodolite Measurement Method 2.2 [35]). Report **sustainable throughput** as the **knee** where these SLOs first break (Karimov Def. 5: "highest load with no continuously increasing event-time latency" [14]), **never a peak-firehose number**.

**Latency definition (Karimov event-time, Def. 1 [14]).** `t0 =` CloudEvent **event-time**; `t_deliver =` commit of the `(event, envelope)` tuple to the entity-partitioned topic such that it is consumable. **Event-time** latency (not processing-time) is mandatory: it includes queue-wait, so measuring only processing-time latency is equivalent to the **coordinated-omission** problem [25] and under-reports the tail exactly under load. For windowed/batch-lane aggregates, set the output's event-time to the **max event-time** of contributing events (Karimov Defs. 3–4 [14]). Report p50/p95/**p99/p99.9**/max via HdrHistogram [24], decomposed into ingest+adapter / route / context-assembly (Graphiti+FalkorDB) / delivery.

**Right-sizing (DS2 [47]).** Set partition count and `keyBy` parallelism from **true processing rate** — records processed per unit of *useful time* (deserialisation + processing + serialisation, excluding blocking on empty input / back-pressured output) — not from CPU or back-pressure, which DS2 shows are inadequate signals. Instrument per-operator useful time to predict the minimal back-pressure-free parallelism per source; this avoids over-provisioning the demand curve. DS2's linear-scaling assumption breaks under **key skew** (hot `subject`s), which §8.5 measures directly.

**SUT-boundary pre-registration (the critical methodological choice).** Theodolite and Karimov isolate the message broker/external store so it is not the artificial bottleneck (the **YSB negative example**: its throughput measured Redis, and windowing was done outside the engine [the broker/KV-store dominated], so it measured the key-value store rather than the stream processor [30]). But CMS context assembly **intrinsically** depends on Graphiti+FalkorDB lookups on the hot path. **Pre-register**: the KG store **is part of the SUT** (and thus part of the demand curve), because removing it would measure a system the thesis does not ship. Keep Kafka tuned (not default — YSB's other flaw was untuned defaults [30]) and **native** (Flink native windowing for the batch lane, not external-store windowing), so the broker is not the artificial ceiling, but report Graphiti assembly latency as a named, first-class component of the latency decomposition.

**Measurement hygiene (Karimov Design Decisions III [14]).** (a) Run the load generator on hardware **separate** from the SUT. (b) **Replay on-the-fly** from the Iceberg/Paimon store, each event stamped with its intended event-time arrival; `latency = delivery_time − intended_arrival_time` (coordinated-omission-safe, wrk2/HdrHistogram style [24, 25]). (c) Measure **throughput at the producer→Kafka boundary** and **latency at the consumer/agent boundary**, never from internal Flink/Kafka counters (engines define throughput/latency differently). (d) Use **25 % warmup**; treat dropped generator connections or unbounded lag growth as a sustainability **FAILURE**. Use Theodolite's monotonicity **search strategies** (lower-bound restriction for the demand metric: once `I` instances sustain workload `W`, the next-larger workload starts its search at `I`) to avoid a full instances × workloads grid [16].

### 8.4 Handling-a-burst metrics (before / during / after)

Four metric families prove a design **handles** a burst [49 (Dean-Barroso), 50, 14, 39]:

- **Tail latency (Dean-Barroso, *The Tail at Scale* [49]).** Use **p99 and p99.9**, not the mean — the slowest few percent cause most of the tail. Report latency-to-agent **during the burst window specifically** (not run-averaged), CO-corrected and decomposed (§8.3); the burst's cost shows up as the **queueing/catch-up term** (`t_queue = lag / drain-rate`), not the steady-state produce term.
- **Lag spike + drain/catch-up (Karimov [14]).** **Lag-spike height** = peak `records-lag-max` during the burst; **drain time** = wall-clock from burst end until `records-lag-max` returns to baseline; **back-pressure duration** = time spent with Flink `backPressuredTimeMsPerSecond` > 0; **watermark-lag** drain. Use **OMB's drain-backlog workload** [13, 50] as the citable harness: `consumerBacklogSizeGB` builds a burst-sized backlog, then consumers drain it and the run ends exactly when the consumer reaches producer parity (it *times* the drain).
- **Recovery after mid-burst failure (Karimov; Breakwater recovery framing [38]).** Use OMB's `producerPauseBeforeDrain` to inject a **TaskManager / consumer-pod kill at peak**; report **recovery time** = wall-clock from kill until the job re-reaches its pre-failure watermark/lag (validates the **Kappa+ replay** claim). Breakwater's reference scale for "no congestion collapse" at a 1.4× spike: recover **< 20 ms** vs DAGOR 500 ms vs SEDA 1.58 s [38] — borrow the *metric* (time-to-restabilise FAST p99 within SLO after a step burst), not the absolute numbers.
- **Graceful degradation / elasticity (BUNGEE + SPEC-RG [51, 52]; brownout/SEDA [40, 39]).** **Goodput must plateau, not collapse**, with FAST holding SLO while BATCH is shed. Report SPEC-RG elasticity metrics: **under-/over-provisioning accuracy** (the area where demand exceeds supply, optimum 0), **reaction time**, and **provisioning timeshare/jitter** [52]; and the **SLO-violation share under surge**. *(Citation precision: BUNGEE is Herbst et al., SEAMS 2015 [51]; the provisioning-accuracy / reaction-time metrics are SPEC-RG metrics [52] that BUNGEE operationalises — attribute accordingly.)*

### 8.5 RQ4 under burst: delivery-layer dynamics

Kafka's **pull + long-poll** model is architecturally burst-robust: producers never block on slow consumers (the durable log decouples burst absorption from burst drain), and a lagging consumer catches up by reading at max rate [53]. Producer-side absorption rests on the page-cache + zero-copy path, batching (`batch.size`, `linger.ms`), and `acks=all` + `min.insync.replicas=2` [54]. Under burst the **dominant failure mode for entity-keyed delivery is hot-key skew** [55]: a burst concentrated on a few hot `subject`s lands disproportionately on a few partitions, whose consumers lag while others idle, and **per-key ordering causes head-of-line blocking** (the Pulsar Key_Shared analogue [55]) — a hot key cannot be parallelised within its key without breaking order. Mitigation: **over-partition**, and for a single dominant key **salt/bucket** (`subject#bucket`), trading strict per-entity order for **per-bucket** order — an honest, *evaluated* trade (measure whether L3 agent utility is affected), not a free lunch. **Over-partitioning is bounded** [56]: excessive partitions add replication latency (~20 ms per 1000 cross-broker partitions) and lengthen controller failover, so partition count must be **swept and the knee located**, not maximised.

**Assignor & semantics.** Use **cooperative-sticky** (KIP-429): lag-triggered autoscaling adds consumers *exactly* when load is highest, and an eager stop-the-world rebalance then would compound the lag spike, whereas cooperative-sticky keeps unaffected partitions processing (measured ~10× lower rebalance pause: 3.5 s vs 37 s in a 10-instance bounce [57]). **At-least-once + agent dedupe** is correct; **duplicates are an expected, measured burst-time artifact** of rebalance reprocessing, not a defect (exactly-once via transactions costs ~10–20 % throughput and ~2–5 ms/txn [58]) — so report **missed events (target 0)** and **duplicates delivered per role + dedupe-store hit-rate**.

**Push-vs-poll under burst (folds H4/E8 into the burst).** Feed **both** the long-poll Kafka consumer (push-like under burst via `fetch.min.bytes` + `fetch.max.wait.ms`) and a fixed-interval poller (sweep 1 s / 5 s / 30 s) the **identical CO-corrected burst trace**, and report the latency/cost/staleness Pareto, attributing the poller's extra latency to **interval staleness** and its cost to **empty-poll waste** between bursts [59] — a single interval is cherry-picking. CO-correction is essential: a lagging poller is precisely the scenario coordinated omission hides [25].

### 8.6 Combined RQ2 + RQ4 experiment and the self-DoS risk

The headline burst result is **H6**: a 2-D frontier of **FAST-lane genuinely-urgent-event p99.9 SLO-attainment vs offered load**, two-lane(rules+anomaly) overlaid on single-lane, with the **two-lane − single-lane gap plotted against B** (maximal at 0.787, → 0 as B → 0 — proving the gain is burst-specific, not free headroom). Frame the design in overload-control vocabulary: the situation-envelope dedup + batch deferral is the **brownout dimmer** (drop optional work to hold the urgent-lane setpoint [40]); the absolute-floor + multi-window guards are **SEDA-style adaptive admission control** [39]; per-entity deviation is the **DAGOR-style priority signal** [37].

**Self-DoS — the signature interaction risk (measure it, don't hide it).** Because a burst *is* often the anomaly, the per-entity detector fast-paths **more** events exactly when load is highest, so the FAST lane can **amplify** rather than absorb the burst. Plot, time-aligned over the burst, **(a) FAST-lane admission rate** and **(b) FAST-lane urgent-event SLO compliance** on the same wall-clock x-axis: self-DoS shows as (a) spiking and (b) collapsing *together*. Then run a **guard ablation** — remove each guard and show SLO compliance degrades — quantifying each guard's load-shedding contribution: the **absolute-volume floor** (Cloudflare "≥ 200" [16]) suppresses novelty-only fast-paths; **multi-window confirmation** (short 5-min AND long ~1–4 h must both cross) requires persistence so a single spike alone does not flood; the **per-window situation-envelope dedup** collapses N raw burst events on one entity into **one** FAST envelope (the anomalous window + top-k most-deviant events) — the principal load-shedding lever, the streaming analogue of Aurora/Borealis window-drop [48].

**Benign-vs-anomalous bursts (proves the detector catches the *right* phenomenon).** Include **two burst archetypes**: a **benign/in-distribution flash crowd** (volume up, mix/entropy stable — correct behaviour: FAST admission stays *low*, ride it on the batch lane) and a **genuine anomalous burst** (mix/entropy shifts, e.g. payment-failure fraction jumps — correct behaviour: the envelope fires once and urgent events make SLO). The discriminator is the **volume-decoupled (ratio/entropy) features** the RQ2 reframe already adopts, validated by the flash-crowd-vs-DDoS literature [60] (a benign flash event raises source entropy but not cluster entropy; an attack raises both) — this is what prevents self-DoS on a legitimate surge.

**Per-entity keyed-state growth under burst (the scalability half, H5).** Report: (a) **max keyed-state size per partition** during burst vs the **hot-entity-LRU cap**; (b) **per-partition `records-lag-max`** (concentrated or spread? = partition-load **Gini**); (c) **LRU eviction rate** and **group-fallback-baseline hit-rate** during burst [16]; (d) **cross-entity fairness index** — do urgent events on **cold** (non-bursting) entities still make SLO, or does the hot-entity burst starve them (cross-entity head-of-line blocking, the DAGOR fairness analogue [37])? The scalability headline: FAST-lane urgent p99.9 SLO-attainment stays **flat** as partitions and entity cardinality grow at fixed per-partition load, and degrades **gracefully** (not cliff) under deliberately skewed hot-entity bursts. **Open tension to sweep:** the hot-entity-LRU cap must bound state growth **without evicting an entity mid-burst** (which would reset its baseline exactly when it is firing — baseline poisoning).

### 8.7 L-context: context-retrieval quality (ContextBench-grounded)

Problem (A) opens the black box of *what context the engine assembled*, grounded in **ContextBench** [61, 62] — a process-oriented, gold-context retrieval benchmark. The envelope `C(e) = {entity_state, recent_events, relevant_facts}` is structurally a **retrieved-context set**, the exact unit ContextBench scores. For an event `e` with human-annotated gold context `G(e)`:

- **Context-Recall**`(e) = |G(e) ∩ C(e)| / |G(e)|`; **Context-Precision**`(e) = |G(e) ∩ C(e)| / |C(e)|`; **F1** = harmonic mean. Edge-case convention (verbatim ContextBench): `|G(e)| = 0 → recall = 1.0`; `|C(e)| = 0 → precision = 1.0`. Report **macro-average** (per-event equal weight) as headline and **micro-average** (Σintersection / Σgold, size-weighted) as secondary — bursty entities dominate the micro-average and could mask poor assembly on rare/cold entities.
- **Three Graphiti granularities** (replacing ContextBench's code-specific file/block/line [verification: granularities are file/block/line, *not* file/symbol/span]) via **deterministic exact-match set-intersection on stable IDs**: **entity-node level** (was the right `subject` referenced?), **fact-edge level** (right typed temporal relation, e.g. `customer:42 -[escalated]-> incident:7`), and **event-tuple level** (the exact recency-windowed `recent_events`). Use node/edge/event **IDs**, not byte-interval overlap; for paraphrased free-text facts add an optional semantic-similarity threshold (RAGAS-style [30]) with a **sensitivity analysis** on the threshold.
- **Context-Precision is a first-class metric, not an afterthought.** ContextBench's central finding is a **recall-over-precision bias** — LLMs over-retrieve to maximise coverage, capping precision/F1 [61]. The thesis's token-waste economics (L3, §5) are *precisely the cost of low context-precision*: cross-reference them — **high envelope recall + low precision should predict high input-token-waste-rate**.
- **Assembly Drop** (the CMS analogue of ContextBench's *Usage Drop*) = `coverage(Graphiti subgraph explored at point-in-time assembly) − coverage(envelope C(e))` — exposes facts that were **retrievable but dropped** under the envelope token/size budget. (Caveat from verification: ContextBench's *efficiency* is a trajectory-ordering metric over a multi-step agent read trajectory; the engine's Graphiti traversal is a **single point-in-time query**, so the "explored" set is the **full candidate subgraph before ranking/budget-cutting**, and "AUC-Coverage" becomes a coverage curve over **budget tiers (top-k)** — does the right fact survive the cut early? — rather than over read-steps.)

**Gold-context construction (ContextBench annotate→verify→minimise loop [61], retargeted).** For a stratified set of outgoing events (oversample boundary / too_urgent / too_slow / FAST-routed, per §2): (i) **annotate** — 2+ annotators trace dependencies from the triggering event (entity links, causal/temporal edges, prior interactions) **as-of `event_time`**, respecting Graphiti bi-temporal valid-time so no **future leakage**; (ii) **verify sufficiency** — replace ContextBench's "patch passes the test suite" oracle with **"an agent conditioned *only* on `G(e)` produces the correct verdict/assist"** (the L3 verdict vocabulary {appropriate, too_urgent, too_slow, irrelevant}); if it fails, add facts and re-annotate; (iii) **minimise** — a *second* annotator removes any fact whose deletion still yields the correct action, giving the precision denominator teeth. Store `G(e)` as ID lists `{node_ids, edge_ids, event_ids}` so the same set-intersection code applies. **Pair each gold context with a gold ACTION**, just as ContextBench pairs gold context with a passing patch. Pre-empt the "this is just a coding benchmark" objection by stating explicitly what does **not** transfer: the coding dataset, repo/AST structure, byte-interval matching, and the test-suite oracle.

**Targets, inter-annotator agreement, and sanity floors.** Primary Lc metric: **Context-F1 at the fact-edge granularity** (where the agent's reasoning lives), macro-averaged, target **≥ 0.70**, with **Context-Recall ≥ 0.85** (a missing needed fact is the costly error) and **Context-Precision ≥ 0.60** as the discriminating floor. Report inter-annotator agreement on the binary "fact ∈ `G(e)`?" decision per granularity — **Cohen's κ** and the pre-adjudication **Jaccard overlap** of the two annotators' gold sets — and trust per-source context-precision only where κ ≥ ~0.6 (the same gate as the L3 judge). The sufficiency oracle is noisier and more policy-dependent than ContextBench's deterministic test suite, so also report **silver (single-annotator) vs gold (adjudicated)** agreement, and reuse the L1 **injected-oracle** events as cheap, leakage-free gold contexts to bound annotation cost. **Sanity floors:** run a *retrieve-everything* (Recall→1, Precision→prevalence) and a *retrieve-nothing* baseline through the full Lc suite; any shipped configuration must beat both on F1, or the threshold is mis-set for this setting.

**H7 under saturation — where (A) and (B) meet.** Run L-context metrics **under the burst** too, reporting a **context-recall-vs-assembly-latency Pareto across load levels**: does the engine return stale/partial context to hit the FAST latency budget when saturated? This coupling — context completeness vs the FAST latency setpoint — is the brownout trade-off [40] applied to context, and is exactly the seam between the two problems. **Compose with external memory benchmarks**: instrument the Graphiti retriever on **LongMemEval** [3] and **LoCoMo** [4] to emit these process metrics (recall/precision/efficiency of retrieved evidence vs each benchmark's gold evidence sessions/supporting facts) **alongside** their native end-to-end QA accuracy, yielding a 2×2 diagnostic (high QA + low context-precision = the LLM compensated for a noisy envelope, *fragile*; low QA + high context-recall = generation failed despite correct context, *not a CMS problem*). Keep **RAGAS** [30] as the LLM-judged groundedness check on free-text facts, distinct from these deterministic ID-set process metrics.
